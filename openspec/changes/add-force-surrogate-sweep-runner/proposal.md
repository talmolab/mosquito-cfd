## Why

The Track B force-surrogate program ([docs/force_surrogate/roadmap.md](../../../docs/force_surrogate/roadmap.md), row #3; GitHub issue [#8](https://github.com/talmolab/mosquito-cfd/issues/8)) needs the stage that **produces the real IB-particle force CSVs** the surrogate trains on. PR2 (`add-force-surrogate-sweep-config`, #5) published the committed 27-config corpus (`examples/prelim_sweep/sweep_manifest.json` + the 27 `inputs/inputs.3d.*` decks); PR4 (`add-force-surrogate-dataset`, #7) published the already-merged extractor/driver `scripts/extract_forces.py`, whose contract resolves each config's CSV at `<input-dir>/<config_name>/IB_Particle_1.csv` ([`scripts/extract_forces.py:82-85`](../../../scripts/extract_forces.py)). What is missing is the **runner** that loops the corpus through the pinned `:fp64` container on a RunAI **A40** allocation and writes each run's IB-particle force CSV to **exactly that per-config layout**, so PR4 consumes it with no glue. PR3 is the remaining blocker for PR5 (train) and PR6 (figure).

This is the **force-only** path (design §9, CC-6): every deck already sets `amr.plot_int = -1` (no field plotfiles), so forces come from the IB-particle CSV only — sidestepping the velocity-in-plotfiles bug entirely. The cluster run is **operator-invoked**; **all runner logic is unit-tested cluster-free** (CC-2): the cluster call is an injected seam, so CI asserts command construction, manifest iteration, output layout, provenance, resume, and the completion check with **no RunAI, GPU, or container**.

## What Changes

### Extends capability: `force-surrogate`

- **New module `src/mosquito_cfd/force_surrogate/runner.py`** (all logic; the cluster call is injected, so the library is pure and cluster-free):
  - `build_run_command(config, *, workspace, container_workspace=DEFAULT_CONTAINER_WORKSPACE, runs_subdir=DEFAULT_RUNS_SUBDIR, wing_vertex=None, iamrex_binary=DEFAULT_IAMREX_BINARY) -> list[str]` — **pure** construction of the per-config RunAI `workspace exec` command wrapping the validated IAMReX launch (`mpirun --allow-run-as-root -np 1 <binary> <deck>`) with the container run cwd `<container_workspace>/<runs_subdir>/<name>` and `wing.vertex` staged into it (the deck's `particle_inputs.geometry_file = wing.vertex` is a **relative** path → must sit in the run cwd). No I/O.
  - `check_completion(csv_path, max_step, *, threshold=DEFAULT_COMPLETION_THRESHOLD) -> Completion` — verifies the IB-particle CSV **exists**, its header equals the 29-column schema (`IB_PARTICLE_COLUMNS`, reused from `dataset.py` — never re-listed), and its data-row count is `>= max_step * threshold`. Flags **missing**, **empty** (header-only), and **short** CSVs as incomplete; a full-length CSV passes. Returns `(complete, rows, reason)`.
  - `run_sweep(manifest_path, output_root, *, docker_digest, timestamp, executor, workspace, resume=True, threshold=DEFAULT_COMPLETION_THRESHOLD, ...) -> list[RunOutcome]` — the sequential orchestrator (D1). **Fail-fast up front:** validates the pinned digest (via PR1's `capture_surrogate_run_metadata`), duplicate config names, and each config's `input_file`/`max_step` presence (the published manifest validator does **not** require these) — all **before** launching anything, so a malformed corpus never wastes A40 time. Then for each config: skips it if already complete (D3 resume), else invokes the injected `executor`, **re-verifies completion** (authoritative over the return code), and writes a per-run `run_metadata.json`. Returns one `RunOutcome(name, status, csv_path, rows, metadata_path)` per config (`status ∈ {"completed","skipped","failed"}`); a **failed** config (executor nonzero **or** short/empty CSV after running) is logged and recorded but does **not** abort the corpus (the driver exits non-zero if any failed).
  - `build_wsl_command(command, *, kubeconfig, runai_binary) -> list[str]` — **pure**, unit-tested wrap of a logical `runai` command into the `wsl -e bash -c "export KUBECONFIG=…; <abs runai> …"` form. Lives in the library so the WSL wrapping is covered/linted; the driver's real executor calls it then the single `subprocess.run` — the only line outside CI coverage.
  - `ExecResult(returncode, stdout, stderr)` + the `Executor` seam (`executor(command, *, cwd) -> ExecResult`); the **library module never imports `subprocess`/RunAI** (asserted on `runner.py`, not the driver — D5).
- **Thin driver `scripts/run_sweep.py`** — argparse over the library; injects the **real** WSL+`runai` executor (the only place `subprocess`/KUBECONFIG/operator paths live). All logic stays in the tested module.
- **`__init__.py`** re-exports the new public API (`build_run_command`, `check_completion`, `run_sweep`, `build_wsl_command`, `ExecResult`, `RunOutcome`, `Completion`).
- **`docker.yml` (CC-1 follow-up, decision D2):** add an `id:` to the **fp64** build step (a fresh id, not colliding with the existing `id: build-args`) and a following step emitting the **full** `ghcr.io/talmolab/mosquito-cfd@<digest>` pin (from `docker/build-push-action`'s `outputs.digest`) to the GitHub job summary, **guarded** `if: …outputs.digest != ''` so the `push:false` dispatch never prints a blank line. Operators copy that pin into `--docker-digest`. The foundation wrapper already *requires* a digest; this makes a pullable one obtainable.

### Resolved decisions (the CC-7-equivalents for PR3 — see `design.md`)

- **D1 — Sequential loop in one long-lived A40 container.** One RunAI workspace (kept alive `; sleep infinity`), looping all 27 configs via `workspace exec` (~65 min wall-clock). One container ⇒ one digest ⇒ one clean provenance record; resumable; no contention for the **A40-only** lab quota. Rejected: one-workspace-per-config (quota cannot fit 27 concurrent; fragments provenance + completion tracking).
- **D2 — Explicit operator `--docker-digest ghcr.io/…@sha256:` arg** (the full pullable pin, identical contract to PR4's `extract_forces.py`); the runner validates it via `capture_surrogate_run_metadata` and threads it into each `run_metadata.json`. Rejected: registry auto-resolve (network/tool coupling, not cluster-free-testable). The `docker.yml` digest-emit step (above) makes the digest obtainable.
- **D3 — Idempotent resume.** A config whose `IB_Particle_1.csv` already passes `check_completion` is **skipped** (no command issued), so a re-run resumes a preempted corpus (A40 is preemptible). Pairs with PR4's `allow_missing`. The skip is **logged**, never silent.
- **D4 — `--output-root` CLI arg; raw CSVs not committed.** Per-config dirs live under the cluster mount (default `examples/prelim_sweep/runs/`). Raw IB CSVs are intermediate and **not git-committed** (`.gitignore` already ignores `IB_Particle_*.csv` globally; this PR adds `examples/prelim_sweep/runs/` so the run tree + per-run metadata stay untracked). The committed corpus + provenance land via PR4's `dataset.parquet` when the real A40 run happens. Per-run `run_metadata.json` records **portable** fields only (config name, the digest, the caller timestamp, the deck's repo-relative path + SHA256, the exact logical command in container paths) — **never** absolute `Z:\` mount paths. This keeps the tested-code PR free of fabricated cluster artifacts (scientific-honesty parity with PR4's D10).
- **D5 — Injected `executor` seam (the honest mock boundary).** `run_sweep` takes `executor(command, *, cwd) -> ExecResult`. Tests inject a fake that **records the exact command** and **simulates the side-effect** (writes a synthetic `IB_Particle_1.csv` of the expected length into the per-config dir); the real driver injects a WSL/`subprocess` executor that runs the RunAI command. Command string + on-disk layout + metadata are asserted; the cluster is never touched.

### Per-config command (logical, portable)

`["runai", "workspace", "exec", <workspace>, "--", "sh", "-c", <inner>]`, where `<inner>` stages `wing.vertex` into the per-config run cwd and runs `mpirun --allow-run-as-root -np 1 <iamrex_binary> <deck>` with cwd = `<container_workspace>/<runs_subdir>/<name>` so IAMReX writes `IB_Particle_1.csv` there (which the mount maps to `<output-root>/<name>/IB_Particle_1.csv`). Container paths are portable (every operator's mount → `/workspace`); the WSL + `KUBECONFIG` + absolute-`runai`-path wrapping is the **driver's** real executor, not the library.

## Impact

- **Spec:** `force-surrogate` capability gains 8 requirements (command construction; output-layout-matches-driver-contract; per-run provenance; completion verification; idempotent resume; cluster-free injected-executor seam / force-only guard; fail-fast sweep input validation; failed-run isolation).

No solver/Dockerfile/dependency changes; no `uv.lock` change (the runner adds no third-party dependency — stdlib only). Runs only on push-to-main/tags, never on PRs.
